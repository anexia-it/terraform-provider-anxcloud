# DNS Zone and Records Import Example

This example demonstrates how to import an existing Anexia Cloud DNS zone AND all its DNS records into Terraform/OpenTofu management using the import functionality.

## How It Works

The `generate-import-blocks.sh` script:
1. 🔍 Discovers your DNS zone via the Anexia Cloud API
2. 📋 Generates import configuration for **both the zone and all records**
3. 🎯 Creates resource stubs with **placeholder values** (not real values!)
4. ⚡ During import, Terraform fetches the **real configuration from the API**
5. ✅ All placeholder values are **automatically replaced** with actual API values

**Important:** You only need to know the zone name (e.g., "example.com"). The script handles everything else automatically, including fetching the real zone configuration from the API.

## Prerequisites

- Anexia Cloud account with an existing DNS zone
- `ANEXIA_TOKEN` environment variable set with your API token
- OpenTofu CLI installed

## Generated Files

This example generates several files during execution. All generated files are prefixed with `_` (underscore) and are excluded from version control:

- **`_import-blocks.tf`**: Import blocks generated by `generate-import-blocks.sh`
- **`_generated.tf`**: Resource definitions created by `tofu plan -generate-config-out`
- **`_plan`**: Plan archive if you use `tofu plan -out=_plan`

These files are user-specific and should not be committed to version control.

## Step-by-Step Import Process

### Step 1: Discover Existing Records

First, run the configuration to discover all existing DNS records in your zone:

```bash
cd examples/dns-zone-records-import
export ANEXIA_TOKEN='your-token-here'
tofu init
tofu apply
```

This will output all discovered records and their import IDs.

**Note:** SOA (Start of Authority) and NS (Name Server) records are automatically skipped during import generation as they cannot be modified through the Terraform provider.

### Step 2: Choose Import Method

You have two options for importing the records:

#### Option A: Manual Import Commands (Recommended for small zones)

Use the output from Step 1 to run individual import commands:

```bash
# Example commands (replace with actual values from output)
tofu import anxcloud_dns_record.www_a 'example.com/<identifier>'
tofu import anxcloud_dns_record.mail_mx 'example.com/<identifier>'
```

#### Option B: Import Blocks (For automation)

Create import blocks in your Terraform configuration for each record:

```terraform
import {
  to = anxcloud_dns_record.www_a_record
  id = "example.com/<record_identifier>"
}

import {
  to = anxcloud_dns_record.mail_mx_record
  id = "example.com/<record_identifier>"
}
```

Alternatively, use the provided script to automatically generate import blocks:

```bash
# Run from the example directory
cd examples/dns-zone-records-import
./generate-import-blocks.sh example.com

# OR run from your own Terraform project
cd ~/my-terraform-project
/path/to/generate-import-blocks.sh example.com
```

This creates `_import-blocks.tf` with import blocks for all records in the zone.

### Automatic Mode (Recommended)

For a fully automated import workflow, use the `--auto` flag:

```bash
# Automatic mode - one command does everything
./generate-import-blocks.sh --auto example.com
```

**What it does:**
1. ✅ Generates import blocks for all DNS records
2. ✅ Automatically generates resource configuration
3. ✅ **Validates plan contains ONLY imports** (safety check)
4. ✅ Automatically applies imports
5. ✅ Creates `imported.tf` with your resource definitions
6. ✅ Cleans up all temporary files

**Safety Guarantees:**
- ⚠️ **Will NEVER create, modify, or destroy resources**
- ⚠️ **Aborts immediately** if plan shows any non-import changes
- ⚠️ **Multiple validation layers** ensure zero risk

**Result:**
After running, you'll have a clean `imported.tf` file containing all your DNS records that you can add to version control and use in your Terraform project.

This is the safest and easiest way to import DNS records!

### Running from Any Directory

The script can now be run from any directory, not just from the example directory:

```bash
# From your Terraform project directory
cd ~/my-terraform-project

# Call the script from anywhere
/path/to/terraform-provider-anxcloud/examples/dns-zone-records-import/generate-import-blocks.sh example.com
```

The script will:
1. Validate no conflicting temporary files exist
2. Create temporary Terraform configuration files (`_temp_main.tf`, `_temp_variables.tf`) in your current directory
3. Generate `_import-blocks.tf` in your current directory
4. Leave temporary files in place for the next steps

**Important:** The temporary files are **NOT automatically removed** because you need them for the subsequent Terraform commands (`tofu plan`, `tofu apply`). After you've completed the import workflow, clean up manually:

```bash
# After completing the import workflow, clean up:
rm _temp_*.tf
rm -rf .terraform/
```

**Note:** The script will abort if `_temp_*.tf` files already exist to prevent conflicts. Remove old temporary files before re-running the script.

### Step 3: Generate Resource Configurations

After creating import blocks, use OpenTofu's config generation feature:

```bash
tofu plan -generate-config-out=_generated.tf
```

This will automatically create the `anxcloud_dns_record` resource configurations in `_generated.tf`.

### Step 4: Apply and Verify

Apply the configuration to bring the records under Terraform management:

```bash
tofu apply
```

## Understanding Resource Stubs and Placeholder Values

⚠️ **IMPORTANT:** The values in `_import-blocks.tf` are **NOT** your real zone configuration!

When you generate import blocks, the script creates resource stubs with **placeholder values**:

```hcl
resource "anxcloud_dns_zone" "zone" {
  name         = "example.com"
  is_master    = true              # ← PLACEHOLDER!
  dns_sec_mode = "unvalidated"     # ← PLACEHOLDER!
  admin_email  = "admin@example.com" # ← PLACEHOLDER!
  refresh      = 3600              # ← PLACEHOLDER!
  # ... etc
}
```

**These are temporary values!** During import:
1. Terraform reads the import block
2. Calls the Anexia Cloud API to fetch **real zone configuration**
3. **Replaces all placeholder values** with actual API values
4. Stores the real values in `terraform.tfstate`

**You do NOT need to:**
- Edit these stubs to match your zone
- Know your zone's actual configuration beforehand
- Worry about the placeholder values being wrong

**The Terraform provider handles everything automatically!**

After import, if you used `-generate-config-out=_generated.tf`, that file will contain the **real configuration** fetched from the API.

## Understanding Import IDs

### Zone Import ID
```
Format: <zone_identifier>
Example: "a94c1c189b924828a80a850d8ee557e0"
```

The script automatically fetches the zone identifier from the API - you don't need to know it.

### Record Import ID
```
Format: <zone_name>/<record_identifier>
Example: "example.com/abc123def456..."
```

- `zone_name`: The DNS zone name (e.g., "example.com")
- `record_identifier`: The record's API identifier

The script automatically discovers all records and their identifiers.

### Import Fails with "Resource already managed"

If you get this error, the record is already managed by Terraform. Remove the import block for that record.

### Record Not Found

- Verify the zone name is correct
- Check that the record still exists in Anexia Cloud
- Ensure the record identifier is valid

### Permission Issues

- Verify your `ANEXIA_TOKEN` has DNS zone access
- Check that the zone exists and you have read permissions

### Record Types

Some DNS record types cannot be imported or modified:

- **SOA Records**: Start of Authority records are managed by the DNS provider and cannot be modified
- **NS Records**: Name Server records are managed by the DNS provider and cannot be modified

These record types are automatically filtered out during the import generation process.
